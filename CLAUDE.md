## 作業プロセス（厳守）

### 1. ドキュメント先行

仕様変更・機能追加・挙動変更を行うときは、コードより前に該当ドキュメントを更新する。

- ユーザー視点の挙動が変わる → `docs/requirements.md`
- 実装方針・設計判断・新たなピットフォールが発生する → `docs/design.md`
- ビルド・CI・リポジトリ構成が変わる → `CONTRIBUTING.md`
- ユーザーが知るべき新機能 → `README.md`

ドキュメント更新と同じターンでコードを書いてもよいが、**ドキュメントの編集を先にコミット可能な状態にしてから**コードに移ること。
ドキュメントが追いついていない PR は受け付けない方針で進める。

### 2. ピットフォール記録

修正中に「なぜこれをこう書かないといけないか」が非自明な事象（=次の人が同じ罠を踏みうる事象）に出会ったら、その場で `docs/design.md` の該当節へ追記する。
記録すべき内容: **何をしてはいけないか / なぜか / 正しい代替手段**。

### 3. 既存ピットフォールの再確認

コードを変更する前に `docs/design.md` の以下の節を最低限見ること。実装上のクリティカルなルールが書かれている。

- ファイル管理 → `(1)` サフィックス問題（`-o` 指定方針、`uniqueDest` 厳守、タイトル取得方法）
- プレイリスト・ファイル選択 → 非同期取得・モーダル直列化・ドラッグ&ドロップ
- Windows でコンソールウィンドウが開く問題 → `applyOSProcAttr` を **すべての** `exec.Command` に適用

### 4. テスト駆動開発（TDD / Kent Beck の Red-Green-Refactor）

ロジックの実装・修正・バグ修正は、ケント・ベックの TDD サイクルに**必ず**従う。「先にコードを書いて後からテスト」は禁止。

**Red → Green → Refactor を 1 サイクルずつ回す:**

1. **Red** — まず失敗するテストを 1 つ書く。期待する振る舞いをテストで表現し、`go test` で**赤（失敗）になることを確認する**。コンパイルエラーや「未実装で失敗」も Red のうち。テストが最初から緑なら、そのテストは何も検証していないので書き直す。
2. **Green** — そのテストを通す**最小限**の実装を書く。きれいさは後回しでよい。`go test` が緑になることを確認する。
3. **Refactor** — テストが緑のまま、重複除去・命名改善・整理を行う。リファクタの各ステップごとに `go test` を回し、緑を保つ。

- 1 サイクルで扱う振る舞いは小さく刻む。大きな機能は複数サイクルに分割する。
- バグ修正も同じ。**まずそのバグを再現する失敗テストを書いて Red にしてから**修正する（修正後にそのテストが緑になることで、回帰テストとして残る）。デグレ防止（次節）と直結する。
- **デッドコードを作らない:** テストが存在しない実装は「呼ばれていれば必ずテストから呼ばれる」状態にならず、未使用に気づけない。すべての実装はまずテストから駆動されて生まれる、を原則とする。

**対象別の方針:**

- 純粋関数（`sanitizeFilename` / `uniqueDest` / `parseYtDlpLine` / `isValidURL` など）は Go 標準テスト（`*_test.go`）で上記サイクルを厳守する。
- 外部プロセス（yt-dlp / ffmpeg）に依存する箇所はモックではなく、**振る舞いの境界を絞った純粋関数を切り出して**テスト可能にし、その関数に対して TDD を回す（例: yt-dlp 出力 1 行 → `parseYtDlpLine`）。
- フロントエンドの UI 変更は自動テストが難しいため手動確認になるが、変更前後で挙動が変わる経路（成功・失敗・キャンセル・並行登録など）を必ず動かしてから完了報告する。これは TDD の代替であって免除ではない。ロジックを Go 側の純粋関数へ寄せられるなら寄せてテストする。

**完了報告には必ず `go test ./...` の実行結果を含める。** テストを書かずにロジックを変更した場合は、その理由を明示すること（原則は「書く」）。

### 5. デグレ防止

過去に直した不具合の再発はユーザーが強く嫌う。再発を疑わせる変更（特にダウンロード処理・ファイル名処理）を入れる前に:

1. `docs/design.md` の該当節を読む
2. 変更が既存ルールに反していないか自問する
3. 反する場合は理由を明示してドキュメント側を先に更新する

「コードだけ直してドキュメントは後で」は禁止。ドキュメントの記述とコードの実態が乖離した瞬間にデグレリスクが立ち上がる。

## コミュニケーション

- ユーザーへの応答は日本語で簡潔に
- ファイル参照はクリック可能な形式（`[name](path)` または `[name](path#L42)`）を使う
- 変更したファイルと意図を末尾でまとめる
